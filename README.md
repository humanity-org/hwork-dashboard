# hwork-dashboard

A static dashboard for the **Hwork / Kaito** project. The source of truth is **GitHub Project #4 (team-platform)**, scoped to the `humanwork` repo, read **read-only**.

## Architecture (why it won't break)

```
GitHub Project #4 ──(read-only GraphQL)──> scripts/build-data.mjs ──> data/issues.json
                                                                          │
data/readiness.json  (curated by a human/agent, structured data only)     │
                                                                          ▼
                              index.html + assets/app.js  (FIXED CODE) ── fetches the JSON, renders
```

- **Presentation is fixed code.** `index.html`, `assets/app.js`, `assets/style.css` are written once and never regenerated. This is the key difference from a setup where an agent rewrites HTML and occasionally corrupts it.
- **Only data files change.** `data/issues.json` is regenerated by a deterministic GitHub Action. `data/readiness.json` is the one file a human or the agent hand-edits (structured JSON, not markup).
- The site reads those JSON files client-side, so a bad data write can at worst show stale/empty panels — it can never break the page structure.

## Files

| Path | What | Who edits |
|------|------|-----------|
| `index.html`, `assets/*` | Presentation | Fixed — humans only, rarely |
| `scripts/build-data.mjs` | Read-only board → `issues.json` | Fixed |
| `.github/workflows/refresh.yml` | Scheduled refresh (every 30 min) | Fixed |
| `data/issues.json` | Auto-generated board snapshot | The Action (never by hand) |
| `data/readiness.json` | Launch checklist | Human / agent (data only) |

## One-time deploy

1. Create repo `humanity-org/hwork-dashboard` and push these files to `main`.
2. **Create a fine-grained PAT** (this is the only credential needed):
   - Resource owner: `humanity-org`
   - Repository access: read for `humanwork` (Contents: Read; Issues: Read)
   - **Organization permissions → Projects: Read-only**
   - No write scopes. Do **not** paste it anywhere except the secret store below.
3. Add it as a repo secret named **`HWORK_DASHBOARD_TOKEN`** (Settings → Secrets and variables → Actions).
4. Enable Pages: Settings → Pages → Source = "Deploy from a branch", Branch = `main`, folder = `/ (root)`.
5. Run the workflow once manually (Actions → Refresh dashboard data → Run workflow) to replace the seed `data/issues.json` with live data.
6. Dashboard is live at `https://humanity-org.github.io/hwork-dashboard/`.

## What the agent (Hwork-watcher) does — and doesn't

- **Does:** optionally edit `data/readiness.json` to keep the launch checklist current. That's structured data only.
- **Does not:** touch any HTML/CSS/JS, and does not need to run the refresh — the Action handles `issues.json` on a schedule. (It may trigger `workflow_dispatch` if you want an on-demand refresh.)
- **Read-only to the board:** nothing here can modify Project #4 or any issue.

## Notes / tuning

- **Priority** is read from issue labels (`P0`/`priority:P0`…) with a title fallback (`[P0]`).
- **Blocked** is read from labels (`blocked`, `blocked:legal`, `blocked:design`, `legal-hold`).
- **Blocked age:** if the board has a `Blocked since` date field, it's used; otherwise the script falls back to last-activity time (shown as `ageBasis` in the JSON). Add a `Blocked since` date field to the board for accurate SLA aging.
- **Needs owner** = open, unassigned, and not in `Backlog`/`No Status`.
- Change cadence in `refresh.yml` (`cron`). Change scope via the `REPO`/`PROJECT`/`ORG` env in the same file.
- To spin up a sibling dashboard for another project, copy this repo and change `REPO` (and the readiness content).
